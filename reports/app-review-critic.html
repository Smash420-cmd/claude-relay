Software Review · Developer Tools

/relay

The tray app that turns Claude Code's usage limits from a wall into a workflow — by waiting for the wall to come down and walking through it for you.

Tested: v0.4.22 · Windows 11 · Price: Free / MIT · Maker: Patrick (Sojournly)
7.5/10
Verge Score

There is a particular flavour of modern frustration reserved for people who pay for an AI coding assistant and then watch it stop mid-sentence because they've "hit their 5-hour limit." /relay is a 700-line Electron app built by one developer to make that frustration somebody else's problem — specifically, the scheduler's.

Anthropic's Claude Code enforces rolling usage caps: a 5-hour session window and a 7-day weekly window. When you hit one, the agent halts, and your half-finished refactor sits there until the clock resets — often at 2 a.m., when you are asleep and certainly not watching a terminal. The conventional answer is to babysit: keep a tab open, notice the reset, paste "continue," repeat. /relay's pitch is that a computer should do that, because a computer is exactly the kind of thing that is awake at 2 a.m.

It is a Windows system-tray app that schedules prompts into local Claude Code sessions, streams their output to logs, and — the headline trick — detects when a session dies on a limit and automatically re-arms itself to resume at the precise moment capacity returns. Close the window; the tray process keeps the scheduler alive. It is a narrow tool with an opinionated thesis, and for the most part it delivers on it.

01The problem, and who's actually buying

Let's be clear about the audience, because it is small and specific. /relay is not for someone who asks Claude to rename a variable twice a day. It is for the power user running long, autonomous, unattended Claude Code workloads — overnight test-bench runs, multi-session build-outs, anything whose natural length exceeds a single 5-hour budget. The design documents are refreshingly honest about this: the "killer feature," repeated almost as a mantra, is carrying a long run across reset boundaries the session itself cannot cross. A Claude Code session is ephemeral and cannot schedule its own future. /relay is the always-on daemon it hands work to.

That framing — Claude as the ephemeral worker, /relay as the persistence layer — is the most intellectually satisfying thing about the product, and it's what separates it from "a nicer cron job." A running session can enqueue its own continuation via a CLI call before it gets cut off, so the handoff is staged, not scrambled. When the limit lands, a fresh session is relaunched at reset to read a checkpoint file and carry on. It's a genuinely clever inversion, and it's the reason this tool exists.

/relay solves a problem Anthropic created and apparently forgot to fix — elegantly, and just a little dangerously.

The Verdict, in one line

02The features that earn their keep

The scheduler

Refreshingly unglamorous. There's no OS cron, no quartz engine, no dependency sprawl — just an internal timer that ticks every 20 seconds and fires anything due (once, at-next-reset, or a specific time). Because the app is always-on via autostart, that's genuinely all it needs, and the restraint is the right call. A pleasant side effect: if your PC was asleep through a scheduled time, the task fires on the next tick after wake rather than being silently lost. Tasks persist to a plain JSON file with atomic tmp+rename writes. No SQLite, no native modules, no build risk. For this data volume that's the correct lazy choice, and the author says as much in a comment.

Auto-resume on limit reset

The marquee feature, and it works through two cooperating mechanisms. First, the executor watches a stopped session's CLI output for limit-reached language. Second — and this is the smarter half — /relay reaches into Claude.ai's own usage API to read the exact reset timestamp, then schedules the resume for that precise moment rather than guessing. There's even a small piece of correct reasoning baked in: if the weekly window is the one pinned at 100%, it waits for the weekly reset, not the sooner 5-hour one. Details like that are the difference between a demo and a tool.

Live usage gauges

Two bars — 5-hour session and 7-day weekly — with reset countdowns, pulled live from Claude.ai. The implementation is quietly the most ambitious code in the project: it prefers an "authoritative" reading captured by a Claude Code status-line bridge, then blends in a token-delta estimate computed from local transcript files for the minutes since that last capture. It's the same technique community tools like ccusage use, executed with care (cache-read tokens deliberately excluded so they don't swamp the gauge). When the live data goes stale it falls back to a pure transcript estimate. It is more engineering than a usage bar strictly needs — but it's the good kind of over-delivery, and it makes the "prime, don't scramble" philosophy real.

The three Claude Code skills

This is where /relay stops being an app you look at and becomes infrastructure you talk to. /relay lets you type "build the payment flow at 9am tomorrow" inside any Claude session and have it parsed into a scheduled task. /relay-autoresume arms the current session so that if it hits a limit, a resume is auto-scheduled — implemented as a watched arm-file plus a watchdog that fires when usage crosses 100%. /relay-listen hibernates a session in a polling loop until queued tasks finish, then reads their logs and acts. The skills are written to ~/.claude/commands/ on every app launch from a single source in main.js, so they self-update silently. Smart, low-ceremony distribution.

Per-task model + effort, and the resume picker

Every task can carry its own model (seven options across current and legacy Opus/Sonnet/Haiku) and effort level, with the UI correctly gating invalid combinations — xhigh only appears for Opus 4.8/4.7, and effort is disabled entirely for Haiku, which doesn't support it. The session resume picker is similarly thoughtful: it scans local Claude transcripts and surfaces each conversation with its first message as a de-facto title, a friendly slug, the working directory, the live git branch (read straight from .git/HEAD), and an "active" flag — plus folder browsing for fresh tasks. These are the touches of someone scratching their own itch and noticing where it still itches.

03Design & UX

A dark, near-black interface with cyan-and-purple accents that wants very badly to be Linear — and mostly pulls it off.

The renderer is hand-written vanilla JS against a tight CSS design system; no React, no bundler, no build step for the UI at all. That's a defensible decision for a single-window app, and the result loads instantly and looks the part: a 56-px draggable topbar, blurred translucent panels, restrained accent colours, a sensible type scale. It reads as a product, not a script with a window.

Where the polish thins out is depth. This is fundamentally a one-window, one-list app — schedule a task, watch the queue, read a log. There's no command palette, no keyboard-first navigation, none of the things that make Linear actually feel like Linear beyond the colour palette; the design outline itself listed those as "Phase 4 — Polish," and Phase 4 is visibly unfinished. For the target user — who lives in a terminal anyway — that's a venial sin. But "Linear-style" is doing some aspirational work in the marketing.

04Reliability

For a personal tool at v0.4, /relay shows more operational paranoia than most shipping software. A single-instance lock prevents a second copy from double-firing every task. A 45-minute hard timeout kills hung child processes so a stuck session can't block the scheduler forever. The tick loop guards against re-entrancy and warns in the log when a tick runs slow. The usage fetch retries with backoff. External writes to the task file — the whole point of letting Claude enqueue its own work — are picked up via a debounced filesystem watcher. This is someone who has been paged at 3 a.m. and built accordingly.

The cracks are where the app meets things it doesn't control. Limit detection leans on a regex against Claude's CLI output — /limit reached|you've hit your|resets? at/ and friends. The day Anthropic rephrases that message, the primary detector goes blind. (The Claude.ai usage API is a sturdier backstop, which softens the blow, but the first line of defence is brittle string-matching against another vendor's UI copy.) "Which session did this task create?" is answered by a heuristic — the most-recently-modified .jsonl — that can pick the wrong conversation if two sessions are active at once. And the JSON store does an unlocked read-modify-write from both the app and the external CLI; the atomic rename prevents a torn file, but two near-simultaneous writers can still clobber each other's task with last-write-wins. Rare in single-user reality, but it's a real edge the SQLite the design called for would have closed.

Testing is the thinnest part of the engineering. There's a syntax check across every file and a smoke test that exercises all 33 model×effort combinations — both good — but no behavioural tests for the scheduler, the executor, the limit parser, or the reset-time math, which is exactly the logic whose silent failure would cost you an overnight run.

05Security posture

Thoughtful where it matters most, and honest about the one place it deliberately leaves the door open.

The standout is cost-safety: before spawning any session, the executor deletes ANTHROPIC_API_KEY from the child environment, by design, so tasks always run on your claude.ai subscription and can never quietly rack up metered API charges. It goes further and scrubs any env var whose name matches KEY/TOKEN/SECRET/PASSWORD/CREDENTIAL — a sensible blast-radius reduction given that headless tasks run with full tool access and could otherwise hand a secret to the model. It's a name-pattern blacklist, not a whitelist, and the code comments say so plainly along with why (a whitelist would break PATH, npm config, proxies). Reasonable trade-off, honestly documented.

Auth is handled the right way: you log in through an embedded claude.ai window, and the sessionKey cookie lives in Electron's own Chromium cookie store — which on Windows is encrypted at rest with DPAPI, scoped to your user account — rather than in some plaintext config the app rolled itself. The cookie is read via the session API and sent only to claude.ai. No token ever touches the task store or the logs.

And then there is the elephant: skipPermissions defaults to on, which means tasks spawn with --dangerously-skip-permissions. An unattended agent runs anything — edit, execute, commit — in the task's working directory with no gate, including prompts that Claude queued for itself. The maker is not hiding this; the flag's name, the code comment, and the docs all spell out the bargain (it's the only way unattended runs complete without stalling on a permission prompt). But "the autonomous overnight agent has root over your repo by default" is a posture, and a more cautious product would ship it off and make the user opt in with eyes open. The secret-scrubbing exists precisely because this door is open — it's mitigation, not a substitute for the door being shut.

One more, more mundane: the installer isn't code-signed (no signing config anywhere in the build), so every user meets a Windows SmartScreen warning on first launch and a "do you trust this" moment from auto-updates. Fine for a tool you built for yourself; a friction wall the moment you hand it to anyone else.

06Weaknesses & what's missing

Pros

  • Solves a real, specific, daily pain — losing work at a usage limit
  • Auto-resume reads the exact reset time from Claude's API, not a guess
  • The skills turn it into infrastructure Claude can drive itself
  • Genuinely clever live-usage gauge (authoritative + token-delta blend)
  • Lean: one runtime dependency, vanilla UI, JSON store, no bloat
  • API-key stripping and secret scrub are deliberate and well-reasoned
  • Cookie stored in DPAPI-encrypted Chromium store, not plaintext
  • Operationally paranoid: single-instance lock, hung-task timeout, retries

Cons

  • Windows-only — no macOS or Linux, no plan for it
  • --dangerously-skip-permissions on by default; full unattended autonomy
  • Limit detection is brittle regex against another vendor's CLI wording
  • Unlocked JSON store can lose a task under concurrent app + CLI writes
  • "Which session?" resolved by a most-recent-file heuristic that can miss
  • No behavioural tests on the scheduler, executor, or reset math
  • Unsigned installer → SmartScreen warnings for anyone you share it with
  • UI is shallow vs the "Linear-style" promise; no palette, no shortcuts
  • Relies on undocumented Claude.ai endpoints that can change without notice

The deeper structural risk isn't in any one line: /relay is a tool built entirely on top of surfaces Anthropic doesn't promise to keep stable — CLI message strings, transcript file formats, an internal usage endpoint, session-registry JSON. It works beautifully against today's Claude Code. It is one upstream refactor away from needing a patch, and that fragility is inherent to the category, not a knock on the craft. The maker clearly knows this; the code is layered with fallbacks for exactly that reason.

07The verdict

A sharp tool for a narrow job, built by someone who needed it

/relay is what happens when a capable developer is annoyed by a specific limitation every single day and decides to fix it properly. It does one thing — carry autonomous Claude Code work across usage-limit resets without you in the loop — and it does that thing with real cleverness: exact reset times pulled from the API, a self-scheduling skill loop, a live-usage gauge that punches above its weight, and a security posture that's careful in the places that matter. The restraint is admirable too; almost nothing here is over-built.

It's held back from a higher score by the things that separate a great personal tool from a great product: a default that hands an unattended agent the keys to your repo, a reliance on brittle string-matching and undocumented endpoints, a thin test net under load-bearing logic, no code signing, and a Windows-only horizon. None are fatal for the power user it's aimed at — and that user will find it close to indispensable — but they're the reasons you'd hesitate before recommending it to someone who isn't already comfortable letting an AI run unsupervised in their working directory.

7.5/ 10 — Very good. A precise fix for a real pain, with sharp edges the target user will happily live with.
