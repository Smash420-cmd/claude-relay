Headline: the 0.4.22 normPct fix over-corrected. The previous review claimed the API returned a 0–1 float; in fact utilization and used_percentage are already 0–100 percentages (tracker.js reads them that way). The new fraction heuristic now mis-scales every genuine reading in the 0–1% range — a real 1% reading becomes 100%, which both lies on the gauge and trips the auto-resume watchdog. One bug fixed, a narrower one introduced. Most other prior findings are genuinely resolved; two of the prior "HIGH"s were overstated and need no change.

High severity — new
high normPct fraction heuristic corrupts the 0–1% range — 1% reads as 100% main.js:25
if (v > 0 && v <= 1) v *= 100 assumes a value ≤ 1 is a fraction. But both source fields are already percentages: the comment itself says utilization is "e.g. 25", and tracker.js:124/138 rounds used_percentage straight to a percent with no scaling. So a user genuinely at 1% usage (utilization === 1) is scaled to 100. Consequences cascade: 1 · gaugerefreshUsage paints the Session/Weekly bar at 100% the moment real usage is 1%. 2 · watchdog false-positivecheckAutoResumeArm (main.js:403) sees sessionPct >= 100, schedules a spurious resume, and deletes the arm file. 3 · 7-day mis-schedulepickResetAt (main.js:61) takes weeklyPct >= 100 as the binding constraint and schedules the resume at the weekly reset — up to 7 days out — when the run was only stopped by the 5h session limit.
Fix
Delete the heuristic. The fields are percentages; there is no fraction case to disambiguate, and 1.0 ("100%") is indistinguishable from 1 ("1%") anyway — so the branch is unwinnable. normPct reduces to Math.min(100, Math.max(0, Math.round(used_percentage ?? utilization))).
Medium severity
medium Autoresume watchdog acts on a single unconfirmed reading and burns the arm file main.js:401–413 false-positive
checkAutoResumeArm fires every 30s. On the first reading where sessionPct >= 100 (or weekly), it schedules a resume and immediately fs.unlinkSync(armFile). There is no second-reading confirmation, so any transient 100% — a real API blip, or the normPct bug above — permanently consumes the arm. If a genuine limit is hit later in the session, the arm is already gone and nothing auto-resumes. The arm is the whole point of the skill, and it is single-use against an unverified signal.
Fix
Require the reading to also carry a real reset timestamp before consuming the arm: gate on pickResetAt(usage) being non-null (a true maxed window always returns one), or require two consecutive ticks at ≥100 before deleting. One extra &&, no new state.
medium Helper duplication survives in the standalone files the dedup pass didn't touch scripts/relay.js:36,38,48,62 · src/tracker.js:17 duplication
The 0.4.2x cleanup centralised the renderer modal, the skill writers, and reset-time resolution — good. But uid(), currentSessionId(), findSessionCwd() and nextSessionReset() still live as independent copies in scripts/relay.js, and currentSessionId() has a fourth copy in tracker.js. These are pure-Node (no Electron), so they don't need the duplication — scheduler.js and sessions.js already export the canonical versions. NoteuserDataDir() in relay.js is justified duplication and should stay: src/paths.js does require('electron') at module top, so the CLI cannot import it. The prior review's "just require paths.js" fix would crash the CLI.
Fix
Move uid + the three session/reset helpers into one Electron-free src/util.js; require it from relay.js, tracker.js, scheduler.js, sessions.js. Leave userDataDir where it is.
Low / acknowledged ceilings
low Secret scrub blacklist misses value-shaped secrets (DATABASE_URL, *_DSN, *_PAT, *_PASS) src/executor.js:88 security · ceiling
The regex strips names ending in KEY/TOKEN/SECRET/PASSWORD/CREDENTIAL/ACCESS_KEY — so GH_TOKEN, AWS_SECRET_ACCESS_KEY, PGPASSWORD are caught. But a connection string in DATABASE_URL (password embedded), *_DSN, a *_PAT, or MYSQL_PASS slip through to a child running with --dangerously-skip-permissions. The ponytail: comment already names this as a known blacklist ceiling, so this is a "raise the wall a little" note, not a defect.
Fix
Add URL|DSN|PASS|PAT|CONN(ECTION)?(_STRING)? to the pattern, or move to a PATH/SystemRoot/APPDATA whitelist when you next touch this path.

Have relay-autoresume.md record the active model/effort into autoresume.json, then pass them through. Or leave it — YAGNI until someone complains.
